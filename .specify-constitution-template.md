# Ansible Sage Development Constitution

**Copy this file to `.specify/memory/constitution.md` after running `specify init`**

---

## Project Mission

Build an AI-powered playbook generation service that creates validated Ansible playbooks from infrastructure events and publishes them to Git repositories for execution by existing automation pipelines.

**Critical**: Ansible Sage is a **generation and publishing service**, NOT an execution engine.

---

## Core Principles

### 1. GitOps First ⚡

**Generate → Validate → Publish (NOT Execute)**

- ❌ Ansible Sage NEVER executes playbooks
- ✅ All playbooks committed to Git for external pipeline execution  
- ✅ Confidence-based branching: High → main, Medium → review, Low → draft
- ✅ Separation of concerns: Generation vs Execution

**Why**: Safety, auditability, integration with existing workflows.

### 2. Safety by Design 🛡️

- ✅ **Validation Required**: Every playbook must pass ansible-lint
- ✅ **Human Review**: Medium/low confidence playbooks require approval
- ✅ **Audit Trail**: Full metadata in Git commits
- ✅ **No Auto-Merge**: Pull requests for non-production branches
- ✅ **Confidence Transparency**: Score calculation documented and visible

**Decision Rule**: When in doubt, require human approval.

### 3. Ansible Best Practices 📚

- ✅ **FQCN Required**: `ansible.builtin.copy`, not `copy`
- ✅ **Idempotency**: All tasks safe to run multiple times
- ✅ **Error Handling**: Use `failed_when`, `block/rescue`, `ignore_errors` appropriately
- ✅ **Security**: No hardcoded credentials, proper file permissions, validate inputs
- ✅ **Naming**: Descriptive task names in imperative mood

**LLM Prompts**: Enforce these in system prompts, not post-generation fixes.

### 4. LLM Provider Agnostic 🔌

**BYOM**: Bring Your Own Model

- ✅ **Provider Abstraction**: `BaseLLMProvider` interface for all providers
- ✅ **No Vendor Lock-in**: Support Claude, OpenAI, Ollama, custom endpoints
- ✅ **Consistent Interface**: Same API regardless of underlying model
- ✅ **Pluggable**: Add new providers without changing core logic

**Architecture**: Provider is injected dependency, not hardcoded.

### 5. Event-Driven Architecture 🔄

- ✅ **Stateless Processing**: Each event processed independently
- ✅ **Classification First**: Determine known/complex/unknown before generating
- ✅ **Confidence Scoring**: Rule-based, transparent, consistent
- ✅ **Async Operations**: All I/O async (LLM calls, validation, Git operations)

**Pattern**: Event → Classify → Generate → Validate → Score → Publish

## Decision Framework

### When Adding a Feature ✨

Ask these questions **before** implementing:

1. ✅ Does it align with GitOps-first principle?
2. ✅ Does it maintain safety guarantees?
3. ✅ Is it provider-agnostic?
4. ✅ Does it follow Ansible best practices?
5. ✅ Is it properly tested and documented?

**If NO to any**: Redesign or reject.

### When Choosing Technology 🛠️

**Prefer**:
- Python 3.11+ standard library
- Established Ansible ecosystem tools (ansible-lint, ansible-core)
- Well-maintained, popular libraries
- Docker/container-compatible solutions

**Avoid**:
- Unnecessary dependencies
- Bleeding-edge unstable tools
- Vendor-locked proprietary solutions
- Tools that execute playbooks directly

### When Handling Events 📨

**Process**:
1. **Classify**: Known, Complex, or Unknown?
2. **Generate**: Use appropriate LLM model tier
3. **Validate**: ansible-lint with auto-fix
4. **Score**: Calculate confidence transparently
5. **Publish**: To appropriate Git branch
6. **Log**: All decisions with context

**Never**: Skip classification or validation.

---

## Non-Negotiables ⛔

### NEVER ❌

- Execute playbooks directly from Ansible Sage
- Skip validation (ansible-lint)
- Auto-merge to main branch without confidence check ≥80%
- Hardcode credentials or secrets
- Generate playbooks without event context
- Log API keys or tokens
- Use `git push --force` on main branch

### ALWAYS ✅

- Calculate and expose confidence score
- Commit to appropriate Git branch based on confidence
- Validate with ansible-lint before publishing
- Log generation metadata (event, model, tokens, latency)
- Provide human-readable recommendations
- Clean up temporary directories
- Handle errors gracefully with proper exceptions

## Confidence Scoring Algorithm 🎯

**Transparent, Rule-Based, No Black Box**

```python
score = 0.0

# Base score from event classification
if known_safe_event: score += 0.3
elif known_event: score += 0.2
else: score += 0.1

# Validation results
if validation_passed: score += 0.4
elif only_warnings: score += 0.2

# Issue penalty
score -= min(len(issues) * 0.05, 0.2)

# Severity adjustment
if severity == LOW: score += 0.1
if severity == CRITICAL: score -= 0.1

# Known event bonus
if event_type in KNOWN_EVENTS: score += 0.2

# Clamp to [0.0, 1.0]
score = max(0.0, min(1.0, score))
```

**Thresholds**:
- High: ≥ 0.80 (80%)
- Medium: 0.50 - 0.79 (50-79%)
- Low: < 0.50 (< 50%)

**Tunability**: Thresholds configurable via environment variables if needed.

## Error Handling Strategy 🔧

### Philosophy

**Fail Fast, Fail Loud, Fail Safe**

- Return structured errors, don't raise in API handlers
- Log with full context (event ID, stack trace, metadata)
- Clean up resources (temp dirs) even on failure
- Never leave system in inconsistent state

### Exception Hierarchy

```python
SageException (base)
├── ConfigurationError
├── LLMProviderError
│   ├── LLMRateLimitError
│   ├── LLMAuthenticationError
│   └── LLMUnavailableError
├── PlaybookGenerationError
├── ValidationError
│   ├── AnsibleLintError
│   └── YAMLValidationError
└── GitPublishError
```

### Error Response Format

```json
{
  "success": false,
  "error": {
    "code": "VALIDATION_FAILED",
    "message": "Playbook validation failed with 3 issues",
    "details": {
      "issues": [...],
      "event_id": "evt-123"
    }
  }
}
```

## Extension Guidelines 🔌

### Adding New LLM Providers

**Steps**:
1. Inherit from `BaseLLMProvider`
2. Implement abstract methods: `generate_playbook()`, `validate_config()`, `get_status()`
3. Add to provider registry in `sage/core/providers/__init__.py`
4. Add configuration example to `.env.example`
5. Write unit tests and integration tests
6. Update documentation

**Example**:
```python
class CustomProvider(BaseLLMProvider):
    name = "custom"
    display_name = "Custom LLM"
    
    async def generate_playbook(self, request: GenerationRequest) -> GenerationResponse:
        # Implementation
        pass
```

### Adding New Event Types

**Steps**:
1. Add prompt template to `sage/core/prompt_templates.py`
2. Classify in `PlaybookOrchestrator` (KNOWN_SAFE or APPROVAL_REQUIRED)
3. Add validation rules if needed
4. Write integration test
5. Document in README and WORKFLOW.md

## Performance Targets 🎯

### Response Times

- Event classification: < 100ms
- Playbook generation (LLM): < 10 seconds
- ansible-lint validation: < 5 seconds  
- Git publishing: < 5 seconds
- **Total end-to-end**: < 30 seconds

### Scalability

- Support 100 concurrent requests
- Handle 10,000 events per hour
- Support repositories up to 1GB
- Cache LLM responses for duplicate events (5 min TTL)

### Resource Limits

- Max playbook size: 10KB
- Max LLM tokens: 4000 output
- Temp directory: Clean up after 1 hour max
- Git clone: Shallow (--depth 1)

## Security Requirements 🔒

### Secrets Management

- ❌ No secrets in code, logs, or Git commits
- ✅ All secrets via environment variables
- ✅ Redact tokens in error messages
- ✅ Use secret scanning in CI/CD

### Input Validation

- ✅ Validate all event payloads with Pydantic
- ✅ Sanitize all LLM inputs
- ✅ Validate Git URLs before clone
- ✅ Check playbook YAML validity

### Access Control

- ✅ API key authentication (optional but recommended)
- ✅ Rate limiting on API endpoints
- ✅ Git credential rotation support
- ✅ Audit logging for all operations
