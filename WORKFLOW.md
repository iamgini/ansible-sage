# Ansible Maya - Workflow Documentation

## Overview

**Ansible Maya generates playbooks and pushes them to Git. It does NOT execute playbooks.**

The system follows a GitOps workflow where generated playbooks are committed to a Git repository for review, testing, and eventual execution through your existing automation pipeline (AAP, Jenkins, GitLab CI, etc.).

## Core Workflow

```
Infrastructure Event
       ↓
Event Classification
       ↓
Playbook Generation (LLM)
       ↓
Validation (ansible-lint)
       ↓
Confidence Scoring
       ↓
Git Publishing (branch based on confidence)
       ↓
[Your CI/CD Pipeline]
       ↓
[Manual Review if needed]
       ↓
[Execution via AAP/AWX/ansible-playbook]
```

## Confidence-Based Branching

Ansible Maya calculates a confidence score (0.0 - 1.0) for each generated playbook based on:

- Event type (known vs unknown)
- Validation results (passed/failed, issue count)
- Event severity
- Automation mode classification

### Branching Strategy

| Confidence Level | Score Range | Target Branch | Action Required |
|-----------------|-------------|---------------|-----------------|
| **High** | ≥ 80% | `main` | Can be deployed directly |
| **Medium** | 50-80% | `review` | Human review before merge to main |
| **Low** | < 50% | `draft` | Testing and validation required |

### Example Git Structure

```
your-ansible-playbooks-repo/
├── playbooks/
│   └── generated/
│       ├── 20260602-143022_disk_full_web01.yml     (main branch)
│       ├── 20260602-145510_service_down_db01.yml   (review branch)
│       └── 20260602-150320_unknown_app02.yml       (draft branch)
```

## Detailed Workflow

### 1. Event Reception

Events can come from:
- Monitoring systems (Prometheus Alertmanager, Nagios, Zabbix)
- Event-Driven Ansible (EDA)
- Direct API calls
- CLI invocation

Example event:
```json
{
  "event_type": "disk_full",
  "description": "Disk usage at 95% on /var partition",
  "host": "web-server-01.example.com",
  "severity": "high",
  "metadata": {
    "partition": "/var",
    "usage_percent": 95
  }
}
```

### 2. Event Classification

The orchestrator classifies events into three modes:

**Auto Mode** (High Confidence)
- Known safe events: `disk_cleanup_tmp`, `log_rotation`, `service_restart_nginx`
- Low severity
- Well-tested patterns

**Approval Mode** (Medium Confidence)
- Known but impactful events: `disk_full`, `service_down`, `high_cpu`
- Higher severity
- Requires review before production

**Collaborative Mode** (Low Confidence)
- Unknown event types
- Complex scenarios
- Experimental playbooks

### 3. Playbook Generation

Uses LLM (Claude, OpenAI, etc.) with:
- Ansible-specific system prompts
- Event-type templates
- Best practices enforcement
- FQCN requirement

Example generated playbook:
```yaml
---
- name: Remediate disk full on web-server-01
  hosts: web-server-01.example.com
  become: yes
  
  tasks:
    - name: Check current disk usage
      ansible.builtin.shell: df -h / | tail -n 1 | awk '{print $5}' | sed 's/%//'
      register: disk_usage
      changed_when: false
      
    - name: Clean apt cache
      ansible.builtin.command: apt-get clean
      when: disk_usage.stdout | int > 80
      
    - name: Remove old log files
      ansible.builtin.find:
        paths: /var/log
        age: "30d"
        recurse: yes
      register: old_logs
      
    - name: Delete old log files
      ansible.builtin.file:
        path: "{{ item.path }}"
        state: absent
      loop: "{{ old_logs.files }}"
      when: disk_usage.stdout | int > 80
```

### 4. Validation

Runs `ansible-lint` with:
- Auto-fix enabled (optional)
- YAML syntax validation
- Best practices checking
- Security linting (via bandit integration)

### 5. Confidence Scoring

Calculates confidence based on:

| Factor | Impact |
|--------|--------|
| Known event type | +20-30% |
| Validation passed | +40% |
| No errors (warnings OK) | +20% |
| Low severity | +10% |
| Known safe event | +20% |
| Each validation issue | -5% |

### 6. Git Publishing

#### Configuration

Set in `.env`:
```bash
GIT_REPO_URL=https://github.com/your-org/ansible-playbooks.git
GIT_MAIN_BRANCH=main
GIT_REVIEW_BRANCH=review
GIT_DRAFT_BRANCH=draft
GIT_TOKEN=ghp_YourPAT
GIT_USERNAME=ansible-maya-bot
GIT_EMAIL=ansible-maya@example.com
```

#### Publishing Process

1. **Clone repository** to temporary directory
2. **Checkout target branch** based on confidence
3. **Write playbook file** with timestamp prefix
4. **Commit with metadata**:
   ```
   Add generated playbook: disk_full_web01.yml
   
   Confidence: high (85%)
   Event Type: disk_full
   Host: web-server-01
   Validation: Passed
   Model: claude-3-5-sonnet
   Generated: 2026-06-02T14:30:22
   
   Generated by Ansible Maya
   ```
5. **Push to remote** branch
6. **Create PR** (optional, for medium/low confidence)

#### API Usage

```bash
# Generate playbook
response=$(curl -X POST http://localhost:8000/api/v1/events/generate \
  -H "Content-Type: application/json" \
  -d '{
    "event_type": "disk_full",
    "description": "Disk at 95%",
    "host": "web-01",
    "severity": "high"
  }')

# Extract details
event_id=$(echo $response | jq -r '.event_id')
playbook=$(echo $response | jq -r '.playbook')
confidence=$(echo $response | jq -r '.confidence_score')

# Publish to Git
curl -X POST http://localhost:8000/api/v1/events/publish-to-git \
  -H "Content-Type: application/json" \
  -d "{
    \"event_id\": \"$event_id\",
    \"playbook\": $(echo $playbook | jq -Rs .),
    \"confidence_score\": $confidence,
    \"metadata\": {
      \"event_type\": \"disk_full\",
      \"host\": \"web-01\"
    },
    \"git_repo_url\": \"https://github.com/your-org/ansible-playbooks.git\",
    \"git_token\": \"ghp_YourToken\"
  }"
```

## Execution Pipeline (Your Responsibility)

After Ansible Maya pushes to Git, **you** control execution:

### Option 1: AAP/AWX Integration

```yaml
# AAP Project pointed at your playbooks repo
# Job Template watches specific branches
# High confidence (main branch) → auto-execute
# Medium/low confidence → manual approval
```

### Option 2: GitLab CI/CD

```yaml
# .gitlab-ci.yml
stages:
  - validate
  - test
  - deploy

validate:
  script:
    - ansible-lint playbooks/generated/*.yml
  only:
    - review
    - draft

test:
  script:
    - molecule test
  only:
    - review

deploy-production:
  script:
    - ansible-playbook -i production playbooks/generated/*.yml
  only:
    - main
  when: manual  # Or automatic based on your risk tolerance
```

### Option 3: Jenkins Pipeline

```groovy
pipeline {
  agent any
  
  triggers {
    pollSCM('H/5 * * * *')  // Check for new commits every 5 min
  }
  
  stages {
    stage('Lint') {
      steps {
        sh 'ansible-lint playbooks/generated/*.yml'
      }
    }
    
    stage('Test') {
      when { branch 'review' }
      steps {
        sh 'molecule test'
      }
    }
    
    stage('Execute') {
      when { branch 'main' }
      steps {
        sh 'ansible-playbook -i inventory playbooks/generated/*.yml'
      }
    }
  }
}
```

## Human Review Process

For medium/low confidence playbooks:

1. **Review PR** in Git repository
2. **Check playbook** for correctness
3. **Test in staging** environment
4. **Merge to main** if approved
5. **Execution triggers** automatically or manually


## Best Practices

1. **Start Conservative**: Begin with `GIT_AUTO_PUBLISH=false`, review everything
2. **Tune Confidence**: Adjust thresholds based on your risk tolerance
3. **Monitor Results**: Track success rate of generated playbooks
4. **Refine Prompts**: Customize templates in `sage/core/prompt_templates.py`
5. **Test in Staging**: Always test review/draft branch playbooks first
6. **Audit Trail**: Git history provides complete audit trail
7. **Rollback Ready**: Git allows easy rollback of problematic playbooks

## Security Considerations

- **Never auto-execute** without validation
- **Review high-impact** operations (database, networking)
- **Limit AAP permissions** to specific inventory groups
- **Use vault** for secrets in generated playbooks
- **Audit all commits** via Git hooks
- **Rate limit** Git pushes to prevent spam

## Troubleshooting

### Playbook pushed to wrong branch
- Check confidence score calculation logic
- Review validation results
- Adjust scoring weights in `orchestrator.py`

### Git push fails
- Verify `GIT_TOKEN` has write permissions
- Check repository URL format
- Ensure SSH keys are configured correctly
- Review branch protection rules

### Playbooks not executing
- This is expected! Ansible Maya only generates and publishes
- Check your AAP/CI/CD pipeline configuration
- Verify branch monitoring is enabled

---

**Remember**: Ansible Maya is a **generation and publishing tool**, not an execution engine. It produces high-quality, validated playbooks and commits them to Git for your existing automation pipeline to handle execution with appropriate controls and approvals.
